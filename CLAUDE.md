## Project Overview

Rio Agent is a native macOS AI programming assistant built with Swift 5.9+ and SwiftUI. It supports multi-provider AI services, 8 built-in tools, multi-agent DAG-based execution, intelligent task planning, Critic-based error self-healing, and local routing capabilities.

**Platform**: macOS 14.0+, Xcode 15.0+, Swift 5.9+

## Critical Project Rules

### Xcode Project Management

**The Xcode project is generated by xcodegen** — never modify `RioAgent.xcodeproj` directly:

```bash
# After adding/removing Swift files
xcodegen generate
```

All project configuration changes must be made in `project.yml`. The `.xcodeproj` file is regenerated from this source of truth.

### Info.plist Embedding

This project uses SPM with `linkerSettings` to embed `Info.plist` into the binary (see `Package.swift`). The plist is NOT bundled separately — it's linked at the `__TEXT/__info_plist` section.

### API Key Storage

- **API Keys**: Stored securely in macOS Keychain via `KeychainManager` with access group `com.rioagent.app`
- **Config Metadata**: Stored in UserDefaults as JSON (provider, baseURL, model name)
- Each `ConfigSet` = one model instance (provider + endpoint + model + API key)

## Architecture

### Execution Modes

Rio Agent has two execution engines:

#### 1. Single Agent (AgentEngine)

Default path for simple to moderate tasks:

1. **Optional Router Intercept** — `RouterService` pre-filters chitchat or routes to specific targets
2. **Task Analysis** — `TaskPlanner` evaluates complexity (simple/moderate/complex/veryComplex)
3. **Tool-Call Loop** — Up to 9999 iterations via `ConversationLoop.run()`
4. **Error Self-Healing** — After 2+ consecutive errors, triggers `CriticService` analysis
5. **Context Management** — Smart token estimation (ASCII/CJK/structured text), auto-compacts old tool outputs

#### 2. Multi-Agent (MultiAgentEngine)

Four-layer pipeline for complex tasks:

| Layer | Name | Responsibility |
|-------|------|----------------|
| Layer 2 | Planner | Orchestrator generates DAG subtask graph with Worker assignments and dependencies |
| Layer 3 | Execution Guild | Wave-based parallel execution (independent tasks run concurrently, dependent tasks wait) |
| Layer 4 | Critic & Verification | PEV (Plan-Execute-Verify) retry loop with Critic failure analysis |
| Synthesis | Result Synthesis | Orchestrator aggregates subtask results into final answer |

**Worker Types**: `search`, `code`, `file`, `general`, `custom` (with custom system prompt)

**Project Context Injection**: `MultiAgentEngine.buildProjectContext()` creates a rich context (working directory, git status, file tree, AGENT.md) injected into every Worker so they are NOT blind to project state.

### Router Service

Optional pre-processing layer with two modes:

- **Generic Routing**: Uses configured AI model for skip/process decisions
- **Qwen3.5-4B Routing**: Local small model with `guided_json` structured output, routes to skip/code_expert/search_agent/data_analyst/chitchat/process

**Critical**: Qwen router MUST disable thinking mode via `chat_template_kwargs: {enable_thinking: false}` to prevent JSON corruption.

### Conversation Loop Unification

`ConversationLoop.run()` is the **shared execution core** for both streaming and non-streaming paths. It abstracts the AI call behind an `AIResponseFetcher` closure and handles:

- Iteration control & cancellation
- Tool call dispatch & result processing
- Error tracking & critic escalation
- Task plan advancement

This eliminates code duplication between `processUserMessageStreaming()` and `processUserMessage()`.

## Tool System

8 built-in tools implementing the `Tool` protocol:

| Tool | Risk Level | Confirmation Required |
|------|------------|----------------------|
| `execute_command` | Classified per command | Per `CommandClassifier` |
| `read_file` | safe | No |
| `write_file` | normal (outside workdir: confirm) | Conditional |
| `edit_file` | normal (outside workdir: confirm) | Conditional |
| `apply_patch` | normal | Yes |
| `search_files` | safe | No |
| `find_files` | safe | No |
| `list_directory` | safe | No |

### Command Risk Classification (CommandClassifier)

Handles complex parsing including pipes, control operators (`&&`, `||`, `;`), and redirections:

- **safe**: Read-only commands (ls, cat, grep, git status/log/diff) → auto-execute
- **normal**: Most commands → user confirmation
- **dangerous**: rm -rf, sudo, curl, wget, kill -9, sed -i → always confirm

The classifier recursively analyzes compound commands and returns the highest risk level found.

## Model Capabilities System

`ModelCapabilities` is a **data-driven pattern-matching database** covering 30+ models:

- Claude (Sonnet/Opus/Haiku 3.x/4.x) — thinking, vision, 200K context
- OpenAI (GPT-4.x, o1/o3) — vision, JSON, up to 1M context for gpt-4.1
- DeepSeek (v3, r1) — r1 has thinking mode, 64K context
- Qwen (2.5, Max) — JSON mode, up to 128K context
- Gemini (1.5, 2.x) — vision, JSON, 1M context

**Pattern ordering matters**: More specific patterns (e.g., `gpt-4.1`) must come before generic ones (e.g., `gpt-4`) to avoid false matches.

## Development Patterns

### Adding a New Tool

1. Create a class in `Tools/` implementing `Tool` protocol
2. Implement `name`, `description`, `parameters`, and `execute(arguments:)`
3. Register in `ToolRegistry.registerDefaultTools()`

### Adding a New AI Service Provider

1. Create a class in `Services/` implementing `AIService` protocol
2. Add case to `AIProvider` enum in `AppState.swift`
3. Add factory branch in `AIServiceFactory.createService()`

### Testing Command Classification

Safety-critical logic — always add test cases to `SafetyRegressionTests.swift` when modifying `CommandClassifier`:

```swift
func testCommandRiskLevel() {
    XCTAssertEqual(CommandClassifier.classify("ls -la"), .safe)
    XCTAssertEqual(CommandClassifier.classify("rm -rf /"), .dangerous)
    XCTAssertEqual(CommandClassifier.classify("git commit"), .normal)
}
```

## Common Gotchas

1. **Streaming Response Parsing**: SSE format uses `data: ` prefix; handle incomplete chunks with buffer accumulation
2. **Context Window Management**: `AgentEngine` tracks token usage per message and auto-compacts when approaching model limits
3. **DAG Execution Wave Logic**: In `MultiAgentEngine.executeSubTasksDAG()`, tasks are grouped by wave — wave N+1 only starts after wave N completes
4. **Router JSON Schema**: Qwen router uses `guided_json` with strict schema enforcement; any prompt that breaks JSON structure will fail silently
5. **Keychain Access Group**: Must match bundle identifier (`com.rioagent.app`) or keychain operations fail
